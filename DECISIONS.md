## Async Content Generation over Synchronous

- When using synchronous processing the main process will get blocked;
- With async processing we create a job and pushed to the queue. The worker picks it from the queue and start processing, this does not block the main process;
- The polling system then polls the server to find the updates in short interval of time;
- With this approach we ensure that the main process work is to handle the creation of job and providing the status of the job; The job is processed by worker(s);
- This type of setup, allows to increase the number of workers when high traffic;

## Cloud Storage over Local Storage

- Scenario: A teacher creates a job request to generate a PPT for the topic "Solar System for grade 8" and the job is created. The worker process on Server A picks the job, generates the content and store it locally. When polling results mention the Job is completed and PPT is generated, the download request is hit to Server B. As the file was locally downloaded on Server A it will fail to find on Server B and result in download failure.
- To solve this problem, I have implemented the Cloud Storage. All the generated files are uploaded to the cloud storage and the download link can be generated from any of the server.
- In this proto-type I have used AWS S3 for the Cloud Storage. The cost of 1GB/per month is around $0.025 that is approximately INR 2.40.

## Logging (File + Database)

- Logging is essential as it provides information of what has happened and when.
- It allows the developers to trace the issue during failures.
- File logs are implemented as a fallback mechanism that is active every-time to ensure that if database logging fails we have another method of logging. HTTP logs are not saved on database, as the count of HTTP logs are vast.
- Database logs are implemented to query logs based on different filters with ease.
- Each HTTP Request has a Unique Trace ID attached to it. This enables to trace all events/actions and link them together from a single HTTP Request.
- KNOWN Limitation: File logs is to be replaced when multiple servers are running because if not done so each server will have its own logs and the logs will not be aggregated at one place.

## Vector Embeddings for Semantic Caching

- When using LLM models, the main goal is to prevent costs for same content generated again.
- This can be prevented by implementing vector embeddings; Vector embeddings convert the text into an array of numbers. Two requests with the similar intent will have similar vector embeddings.
- If the similarity of the vector embeddings crosses a defined threshold (in our case 0.92), we return the cached output.
- I have used the same provider (Google) for embedding model to keep the setup simpler. This ensures one API Key, one console account and billing account.

## Metrics Dashboard

- I believe the assignment is fundamentally a cost problem. Reducing costs without affecting the quality is very important.
- The metrics dashboard makes the cost impact visible in real time — total spend, cache hits, cost saved by caching, average generation time, and failed jobs.
- This metrics dashboard helps to verify that the optimizations are actually working and the numbers are from the running prototype not estimates.

## Optimizations Implemented

- Async job queue with status polling
- Semantic Caching (Request Deduplication)
- Rate limiting — 10 PPT generation requests per IP per 10-minute window. Prevents a single teacher or a bad actor from flooding the queue and driving up costs.
- Smart model routing — The primary model (Gemini 3.1 Pro) is tried first. If it returns a 503 error, it retries twice before switching to the fallback (Gemini 2.5 Flash). The fallback uses a shorter, more directive prompt designed for smaller models. This ensures that quality is not compromised when things work fine while still having a working fallback.
- Template-first generation — Content (slide titles and bullets) is generated by the LLM separately and then injected into the file structure during PPTX build. The LLM is not involved in layout or formatting at all. This keeps the prompt focused and output predictable.
